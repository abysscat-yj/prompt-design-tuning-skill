# Prompt 设计调优最佳实践

这个 Skill 的目标，不是陪用户“聊 prompt”，而是把 prompt 调优变成一个**可执行、可复盘、可控成本**的工程流程。

Agent 负责绝大多数执行工作。  
人类只负责把关方向、批准高成本循环、最终验收上线版本。

---

## 何时使用

当用户有以下需求时，使用这个 Skill：

- 需要从 0 到 1 设计或优化一个目标 prompt
- 需要单独设计一个机评 / judge prompt
- 需要用评估集比较多个模型效果
- 已经具备 API curl、SDK 调用方式或请求协议
- 希望在有限预算内做可控的 prompt 迭代
- 希望把 prompt 调优流程沉淀为可复用的方法，而不是一次性聊天

---

## 工作模式

### 1. 仅设计模式

适用于：

- 还没有可运行环境
- 还没有评估资源
- 暂时不能真的发起模型请求

此模式下，Agent 输出：

- 任务定义
- 目标 prompt 草案
- 机评 prompt 草案
- 评估方案
- 脚本骨架
- 人工执行建议

### 2. 执行模式

适用于：

- 已有可运行环境
- 已提供模型调用方式
- 已提供评估集、资源限制、候选模型等信息

此模式下，Agent 继续完成：

- 批量生成
- 机评
- 结果分析
- Prompt 迭代
- 最终候选版推荐

---

## 基本原则

以下规则默认不可破：

1. **目标 prompt** 和 **机评 prompt** 必须分离。  
   不允许在同一轮对比中同时偷偷改两者，再把收益混在一起。

2. 大规模评估前，必须先冻结 **任务定义（task spec）**。

3. 每一轮 prompt 优化都必须有明确的 **优化假设**。  
   不能“感觉这句不顺眼，改改试试”。

4. 必须维护 **实验记录**，至少包含：
   - 版本号
   - 本轮改动摘要
   - 改动假设
   - 指标结果
   - 成本信息
   - 结论

5. 高成本循环前，必须经过人工批准。

6. 最终上线候选版，必须经过人工审核。  
   机评高分不等于可以直接上线。

7. 如果输入信息不全，可以做低风险假设，但必须明确写出假设内容。

## 建议收集的输入

Agent 应尽量收集或推断以下信息：

- 业务目标
- 用户场景
- 输入形式
- 输出形式
- 硬性约束
- 不可接受的错误
- 成功标准
- 线上验收门槛
- 评估集
- 候选模型
- 请求方式（curl / SDK / API）
- 资源限制（TPM、RPM、超时、预算、重试上限）

## 人类闸门（Human Gates）

默认只在以下关键节点需要人工确认：

### Gate A — 冻结任务定义
确认：
- 任务理解是否正确
- 成功标准是否合理
- 约束是否完整

### Gate B — 确认目标 Prompt 方向
确认：
- 目标 prompt 是否方向正确
- 是否已经达到“可进入评估”的程度

### Gate C — 确认机评 Prompt 方向
确认：
- 机评标准是否公平
- Judge 是否在评真正想评的东西

### Gate D — 批准开启高成本循环
确认：
- 模型列表
- TPM预算
- 循环轮数
- 是否值得继续烧资源

### Gate E — 最终验收
确认：
- 当前最优版是否可以作为上线候选
- 是否继续优化
- 是否停止

除非用户明确要求更细粒度控制，否则不要在中间频繁打断。

# 执行流程

## Phase 0 — 任务定义（Task Spec）

在写任何 prompt 之前，先建立清晰的任务定义。

任务定义至少应包括：

- 问题描述
- 输入格式
- 输出格式
- 业务目标
- 用户目标
- 约束条件
- 明确禁止的输出
- 正反例
- 成功指标
- 仍未解决的问题
- 当前假设

如果用户描述不完整，不要原地发呆。  
先补出合理假设，再列给用户确认。

完成后，进入 **Gate A**。

---

## Phase 1 — 生成目标 Prompt 初版

根据任务定义，产出第一版目标 prompt。

要求：

- 指令清晰
- 约束显式
- 输出结构稳定
- 尽量减少歧义
- 优先提高可控性，而不是堆很多“风格化”废话
- 示例只在真正有帮助时加入

同时输出：

- 关键设计思路
- 预测的风险点
- 可能失败的场景
- 第一轮评估最值得重点观察什么

完成后，进入 **Gate B**。

---

## Phase 2 — 生成机评 Prompt 初版

设计独立的 Judge / Eval Prompt。

要求：

- 评的是任务结果，不是 prompt 写得漂不漂亮
- 各评分维度拆开打分，再聚合
- 必须包含一票否决类错误
- 输出必须是结构化 JSON
- 尽量减少模型风格偏好造成的偏差
- 明确处理以下情况：
  - 部分正确
  - 格式错误
  - 理解偏差
  - 内容危险或违规
  - 题意不完整导致的合理不确定性

同时输出：

- 评分维度
- 权重设计
- 硬失败条件
- Judge 输出 JSON Schema
- Judge 盲区说明

完成后，进入 **Gate C**。

---

## Phase 3 — 设计评估方案

在真正跑大批量前，先把评估设计清楚。

评估方案至少要定义：

- 评估集来源与规模
- 样本切片方式（简单 / 中等 / 困难 / 边界）
- 线上可接受阈值
- 主指标
- 次级诊断指标
- 平局时如何决策
- 最大循环次数
- 总预算限制
- 提前停止条件

默认循环策略：

- 默认最多只做 **2 轮高成本优化**
- 如果收益很小且失败类型没有本质改善，应停止
- 如果 Judge 本身表现可疑，应优先修 Judge，而不是继续改目标 Prompt

---

## Phase 4 — 编写生成脚本

如果存在可执行条件，Agent 应编写批量生成脚本。

脚本应尽量满足：

- 支持 jsonl / csv / excel 输入
- 支持多个模型
- 支持断点续跑
- 支持重试与退避
- 支持日志记录
- 支持保持输入输出顺序一致
- 支持按 TPM / RPM 做限流
- 生成结构化结果供后续机评使用

### TPM 处理原则

不要把 TPM 直接粗暴换算成大并发。

应优先采用：

- 估算单请求 token 消耗
- 使用 token bucket 或时间窗口限流
- 在 RPM 和延迟未知时，采用保守并发
- 先保证不爆资源，再追求速度

---

## Phase 5 — 批量生成模型结果

对所有指定模型、指定 prompt 版本，跑完整个评估集。

至少记录：

- 模型名
- prompt 版本
- 输入样本 ID
- 原始输出
- token 使用量（若可得）
- 延迟
- 重试次数
- 请求失败情况
- 截断 / 解析失败情况

如果生成过程失败较多：

- 先区分是基础设施问题还是 prompt 问题
- 在排除配额、网络、限流、接口协议问题前，不要贸然判 prompt 不行

---

## Phase 6 — 执行机评

用 Judge Prompt 对生成结果做批量评估。

要求：

- Judge 输出结构化 JSON
- 原始 judge 结果可回溯
- 统计总分和分切片指标
- 自动识别主要失败簇
- 区分格式错误和内容错误
- 如果 Judge 噪声很大，应显式指出，而不是假装结果可靠

---

## Phase 7 — 分析与优化

只有在存在明确优化假设时，才允许开启新一轮 prompt 迭代。

每一轮必须完成：

1. 总结上一轮结果
2. 找出主要失败簇
3. 提出本轮优化假设
4. 只修改最必要的 prompt 片段
5. 给出版本差异摘要
6. 预测本轮应改善什么、可能退化什么

禁止无理由“再来一轮试试”。

如果本轮需要消耗明显资源，先进入 **Gate D**。

---

## Phase 8 — 最终推荐

当某一版已经达到较强水平时，Agent 应输出最终评审包。

至少包括：

- 最终目标 prompt
- 最终机评 prompt
- 推荐模型
- 总体指标
- 分切片指标
- 仍然存在的主要失败类型
- 成本 / 延迟说明
- 是否建议上线
- 上线后建议监控哪些点

完成后，进入 **Gate E**。

# 各 Gate 默认输出内容

## Gate A 输出

- 任务定义文档
- 当前假设
- 缺失信息
- 建议验收标准

## Gate B 输出

- 目标 prompt v1
- 设计 rationale
- 预期风险

## Gate C 输出

- 机评 prompt v1
- 评分 rubric
- JSON schema
- Judge 盲区

## Gate D 输出

- 当前结果对比
- 失败分析
- Prompt 改动摘要
- 下一轮优化假设
- 预计资源消耗

## Gate E 输出

- 最终候选版
- 为什么它是当前最优版
- 它还会在哪些场景出错
- 建议上线 / 继续优化 / 停止

# 默认分析模板

## 实验记录字段

每轮实验至少记录：

- iteration
- production_prompt_version
- judge_prompt_version
- model
- dataset_version
- hypothesis
- change_summary
- aggregate_score
- slice_scores
- dominant_failures
- cost
- verdict

## 失败分类建议

Agent 应尽量把失败归类到以下类别之一：

- 任务理解错误
- 约束遗漏
- 抽取错误
- 推理错误
- 覆盖不完整
- 危险 / 违规
- 格式 / Schema 错误
- 冗长 / 啰嗦
- 幻觉细节
- Judge 与真实目标不一致
- 基础设施失败

---

# 明确禁止的反模式

不要做这些事：

- 同一轮既改目标 prompt 又改 judge prompt，却不说明
- 只看总分，不看失败分布
- 在超小评估集上过拟合，却不提醒风险
- 用机评替代人工最终验收
- 分数有一点点抖动就无限循环
- 一个点坏了，却把整个 prompt 全重写
- 隐瞒关键假设
- 没展示硬例子就宣布“效果很好”

---

# Skill 触发后的默认行为

当该 Skill 被触发时，Agent 应按以下顺序执行：

1. 构建或刷新任务定义
2. 判断当前处于哪一个阶段
3. 优先补齐缺失产物，而不是先重写已有内容
4. 优先做增量优化，而不是全文推翻
5. 只在人类闸门处请求确认
6. 每完成一轮关键动作后，输出一个简洁决策摘要，包括：
   - 改了什么
   - 为什么改
   - 哪些指标改善了
   - 还剩什么主要问题
   - 是否值得继续下一轮

# 示例触发语

以下请求适合触发本 Skill：

- “帮我把这个 prompt 调优流程自动化”
- “先写目标 prompt，再写机评 prompt，再设计评估”
- “根据评估集和几个模型，找出当前最优 prompt”
- “控制预算做 1 到 2 轮 prompt 迭代”
- “把 prompt 调优流程沉淀成可复用的 agent skill”
- “让 agent 自动推进，人只在关键节点确认”
